# Session Similarity Search

## Summary

A hybrid search feature that answers "have I done this before?" by combining keyword search (FTS5) with semantic search (message-level embeddings) and LLM-powered query expansion, then aggregating results at the session level.

## Motivation

After hundreds of sessions across dozens of projects, developers repeatedly solve similar problems without knowing it. The existing search finds individual message hits by keyword, but can't surface the insight: "you implemented pagination 3 weeks ago in jellico-prototyping — here's that session."

Two gaps exist:
1. **Session-level ranking** — current search returns individual messages, not "which sessions are most relevant"
2. **Semantic understanding** — searching "pagination" won't find sessions where you discussed "infinite scrolling" or "load more"

## Architecture: Three-Layer Hybrid Search

### Layer 1: FTS (Keyword Search)

Uses the existing SQLite FTS5 infrastructure. Searches across three scopes in parallel:
- **Messages** — what the user and assistant discussed
- **Tool Inputs** — file paths, grep queries, bash commands, code written
- **Tool Results** — command output, file contents read

Each scope is weighted: tool_input (1.5x) > message (1.0x) > tool_result (0.5x). Tool inputs reflect what the user *did*, making them a stronger relevance signal. Tool results are noisy (full file contents, command output).

**Strengths:** Exact term matches, technical identifiers, file paths, code patterns.
**Weakness:** No synonym understanding — "pagination" won't find "infinite scroll."

### Layer 2: Semantic Search (Message-Level Embeddings)

Embeds individual user messages using ChromaDB with a local sentence-transformer model. Each user message becomes a vector; queries are embedded and compared via cosine similarity.

**Why message-level, not session-level:** We prototyped both. Session-level embeddings compress thousands of messages into one vector, diluting the signal. Results:
- Session-level: "dark mode" vs "night theme" overlap = 2/10 sessions
- Message-level: "dark mode" vs "night theme" overlap = 6/10 sessions

A short query matches much better against individual short messages than against a single massive document.

**Strengths:** Finds semantically similar messages using different words.
**Weakness:** Bare technical keywords ("pagination") don't embed well — the model needs natural language context.

### Layer 3: Query Expansion (Local LLM)

Before running semantic search, the query is expanded into alternative phrasings using a local LLM (Ollama). This bridges the gap between bare keywords and the semantic search layer.

**Prompt (no few-shot examples needed):**
```
List 6 alternative phrases a software developer might use when
discussing "{query}". Comma-separated only, no explanation.
```

**Expansion examples (generated by qwen3:8b, no examples in prompt):**

| Query | Expansions |
|-------|-----------|
| "pagination" | server-side pagination, client-side pagination, infinite scroll, page-based navigation, offset and limit, chunked loading |
| "dark mode" | dark theme, night mode, inverted colors, dark UI, low-light mode, dark interface |
| "error handling" | exception management, robustness, fault tolerance, recovery mechanisms, graceful degradation, defensive programming |
| "deploying to production" | pushing to live, going live, putting into production, launching the release, rolling out to production, making it live |
| "authentication" | user verification, identity confirmation, access validation, credential validation, login process, secure login |

Each expansion is run as a separate semantic search query. Results are merged by session — a session matching multiple expansions scores higher.

**Impact on results (tested on 500 sessions, 3,434 user messages):**

| Query | Semantic sessions (no expansion) | Semantic sessions (with expansion) |
|-------|----------------------------------|-------------------------------------|
| "pagination" | 7 | 27 |
| "dark mode" | 10 | 17 |
| "error handling" | 9 | 37 |
| "deploying to production" | 8 | 37 |

**Cost:** ~1-2 seconds for the LLM call (runs locally via Ollama), plus ~50ms per expansion query against ChromaDB. Total added latency: ~2 seconds.

### Fusion: Reciprocal Rank Fusion (RRF)

FTS and semantic search return independently ranked lists. RRF merges them without needing to normalize scores across different systems:

```
RRF_score(session) = Σ 1/(k + rank_in_fts) + Σ 1/(k + rank_in_semantic)
```

Where `k` is a constant (typically 60). A session appearing in both lists ranks higher than one in either alone.

## Prototype Results

Tested against 500 sessions across 50+ projects.

### "pagination" — the hardest case

| Rank | Project | Sources | Key Match |
|------|---------|---------|-----------|
| #1 | claude-code-analytics | FTS:54, Sem:0.77 | Matched via: infinite scroll, paging through results, next page, load more button |
| #2 | jellico-prototyping | FTS:34, Sem:0.46 | "multiple view modes (Flat/Grouped/Timeline)" — actual pagination implementation |
| #3 | jellico-prototyping | FTS:30, Sem:0.43 | "It still feels sluggish with thousands of rows in flat view" — found via "infinite scroll" expansion |

Without expansion, semantic search returned only 7 sessions (mostly noise). With expansion, it found 27, including sessions about infinite scrolling and cursor-based paging that FTS would never match.

### "deploying to production" — semantic bridging

| Rank | Project | Sources | Key Match |
|------|---------|---------|-----------|
| #1 | claude-code-analytics | FTS:10, Sem:0.49 | "Let's push the changes" — matched via "pushing to production" |
| #2 | lxr | FTS:3, Sem:0.53 | "let's continue with the deploy" — matched via "rolling out to production" |
| #3 | lxr | FTS:2, Sem:0.50 | "How easy is this to deploy to Digital Ocean?" |

### "error handling" — found related infrastructure work

Expansion found a session about "logging/observability improvements" via the "graceful degradation" expansion — a session about error handling infrastructure that never uses the word "error."

### Where each layer contributed

| Layer | What it found that others missed |
|-------|--------------------------------|
| FTS | File paths (`pagination.tsx`), grep queries (`offset.*limit`), exact code patterns |
| Semantic | "you have to scroll back up when you want to change pages" for query "pagination" |
| Query expansion | Bridged "pagination" → "infinite scroll", "deploy" → "going live", "error handling" → "graceful degradation" |

## API Design

### Endpoint

```
GET /api/sessions/similar?q=deploying+to+production&limit=10&exclude_session=<id>
```

### Response

```json
{
  "query": "deploying to production",
  "expansions": ["pushing to live", "going live", "rolling out to production"],
  "results": [
    {
      "session_id": "abc123...",
      "project_name": "/Users/.../lxr",
      "score": 0.0292,
      "fts_hits": 3,
      "semantic_best": 0.53,
      "start_time": "2026-02-15T10:00:00Z",
      "message_count": 175,
      "tool_use_count": 45,
      "sample_matches": [
        {
          "source": "semantic",
          "similarity": 0.53,
          "message_index": 463,
          "text": "let's continue with the deploy",
          "matched_via": "rolling out to production"
        },
        {
          "source": "fts",
          "message_index": 107,
          "text": "How easy is this to deploy to Digital Ocean?"
        }
      ]
    }
  ],
  "total_sessions": 15
}
```

### Parameters

- `q` (required): Search query — natural language or keywords
- `limit` (optional, default 10): Max sessions to return
- `exclude_session` (optional): Session ID to exclude from results

## Frontend Integration

### Approach: Sessions Tab on Existing Search Page

Rather than a separate page, session similarity search lives as a new **Sessions** tab on the existing Search page alongside **All | Messages | Tool Inputs | Tool Results**.

Same search input, same mental model ("I want to find something"). The tab changes what comes back — individual messages vs. ranked sessions.

Results render as session cards showing:
- Project name, session date, message/tool count
- Up to 5 matching messages with deep links to `#msg-{index}`
- Which query expansions matched
- Score breakdown (FTS vs semantic)

**Progressive loading**: FTS results appear instantly (~200ms), then semantic + expansion results merge in (~2s) for a seamless experience.

## Implementation Plan

### Phase 1: FTS Session Search (no new dependencies)

- New router: `claude_code_analytics/api/routers/similar.py`
- Reuses existing `DatabaseService` search methods
- Session-level aggregation and scoring
- Provides useful results immediately for exact keyword matches

### Phase 2: ChromaDB + Message Embeddings

- Add `chromadb` to dependencies
- New service: `claude_code_analytics/services/embedding_service.py`
  - Manages ChromaDB collection lifecycle
  - Batch embed existing messages on first run (~70s for 500 sessions)
  - Incremental embed on import via file watcher
- ChromaDB runs in-process, stores data in `~/.config/claude-code-analytics/chroma/`

### Phase 3: Query Expansion + Hybrid Fusion

- Integrate with Ollama for local LLM query expansion
- RRF fusion of FTS and semantic results
- Frontend page at `/similar` with search input and session cards
- Add to sidebar navigation
- Falls back gracefully: if Ollama unavailable, skip expansion; if ChromaDB empty, FTS-only

### Phase 4: Session Detail Integration

- "Related Sessions" panel on session detail page
- Auto-query generation from session content

## Dependencies

| Phase | New Dependencies | Notes |
|-------|-----------------|-------|
| Phase 1 | None | Uses existing FTS |
| Phase 2 | `chromadb` (~50MB) | Includes sentence-transformers, runs in-process |
| Phase 3 | Ollama running locally | Already supported for analysis; optional — degrades gracefully |

## Open Questions

1. **Embedding model**: Start with ChromaDB's default (all-MiniLM-L6-v2) or configure a better model via Ollama? The default handles natural language well but struggles with bare technical keywords.

2. **Query expansion caching**: Cache LLM expansions for repeated queries? A simple dict cache would avoid redundant Ollama calls.

3. **Should assistant messages be embedded too?** Currently only user messages. Assistant messages would add recall but 3x the vector count and include boilerplate.

4. **Short query handling**: For single-word queries, should we require natural language for semantic search and fall back to FTS-only? Or always expand?
